# V0

`special specs` is the canonical product contract for this repo. This document is rollout framing and operator-oriented narrative only; exact shipped behavior should live in `specs/` plus its verifies, not here.

## Goal

`projector` v0 should be good enough for one person to dogfood across their active repos and machines.

That means:

- one or more repo-local gitignored mounts can be materialized on demand
- UTF-8 text files inside those mounts stay synced across checkouts
- agents can read and edit the local files directly
- overlapping edits from the same person across machines or agents converge reliably
- there is enough provenance to understand what changed and where it came from
- private project context stays co-located with code in the local checkout without entering the repo's VCS flow

v0 is not trying to solve general team collaboration yet.

## Primary dogfooding user

The primary v0 user is a single owner with:

- multiple product repos
- multiple local checkouts
- multiple editing actors such as terminal agents, editors, and scripts
- a need for private strategic context that should not live in the product repo

The practical target is:

- 3 to 10 repos
- 2 to 4 active machines or workspaces
- 1 human identity
- several agent identities under that human

## Core v0 workflow

1. User installs `projector` and either deploys or connects to one or more projector servers.
2. They run `projector sync start` during setup or login so projector is always syncing on that machine.
3. In a product repo, they use `projector add <path>` to start syncing one or more local files or folders, or `projector get` to materialize an existing remote sync entry into the checkout.
4. Humans and agents edit plain text files inside those mounts.
5. The machine-global daemon watches local edits and pushes them upstream.
6. Remote changes arrive quickly and are written back to disk.
7. The user can inspect recent changes and sync state when something looks off.

If this workflow feels boring and dependable, v0 is successful.

## Must-have scope

### Repository and workspace model

- one or more repo-local sync-entry attachments per repo projector configuration
- each sync entry binds one whole local file or folder to one whole authoritative remote sync entry through one server profile
- no partial attachment to subsets of an existing remote sync entry
- local gitignored path materialization rooted in the repo
- server-first source of truth

### File types

- UTF-8 text only for v0
- nested directories allowed under configured mounts
- UTF-8 text only

### Lifecycle operations

- create file
- edit file body
- delete file
- create directory as needed from manifest paths

## Deliberate v0 cuts

These should be excluded unless they are required for the dogfooding loop:

- arbitrary repo file sync outside configured mounts
- binary assets
- true multi-user sharing across a team
- comments, reactions, or discussion threads
- live presence or "who is here right now" features
- editor plugins
- mobile or browser-first editing

### Rename and move policy

v0 should avoid path cleverness.

Support explicit path updates in the manifest if the implementation is straightforward, but do not block v0 on rename detection. If local rename handling becomes noisy, treat it as delete plus create in v0 and revisit once the core loop is stable.

## User-visible product surface

v0 should stay small:

- `projector sync start`
- `projector sync stop`
- `projector add`
- `projector get`
- `projector remove`
- `projector rm`
- `projector status`
- `projector log`
- `projector doctor`
- `projector restore`
- `projector connect`
- `projector deploy`

### `projector sync start`

Starts the machine-global sync daemon and keeps bound repos on that machine reconciled continuously. This is the normal always-on mode and should be easy to add to startup during setup.

### `projector sync stop`

Stops the machine-global sync daemon cleanly.

### `projector add`

Registers a repo-local file or folder in projector configuration, bootstraps that whole local sync entry against the chosen connected server profile immediately, and then keeps it synced through the machine daemon. If the target is already under version control, projector should warn and require `--force`.

### `projector get`

Discovers existing remote sync entries on a chosen connected server profile and materializes one whole remote sync entry locally. The low-level form accepts a stable server-side sync-entry id plus an optional repo-local destination path; if omitted, projector uses the remote entry path as the local materialization path. The normal human flow uses a terminal browser with source repo metadata and content preview.

### `projector remove`

Removes a synced file or folder from projector configuration so that local path stops syncing to its remote counterpart.

### `projector rm`

Alias for `projector remove`.

### `projector status`

Shows:

- bound workspace
- daemon state
- last sync time
- pending local changes if any

### `projector log`

Shows recent durable events with actor, path, timestamp, and summary.

### `projector doctor`

Runs an explicit setup and diagnostics pass for the current repo: connected-profile reachability, repo registration, per-entry gitignore and tracked-file sanity, runtime lease state, and recent sync issue state. This is the audit surface; `projector status` stays compact.

### `projector restore`

Opens a terminal revision browser for one projected text document, shows diffs against the current local file, and asks for confirmation inside the browser before applying a selected restore. Restore is append-only: applying an older revision makes that content current again as a new head revision rather than truncating later history. Low-level `--seq` restore remains available as a scripted escape hatch.

### `projector connect`

Attaches the local machine to one or more known projector servers, persists named server profiles in machine-global projector state, and lets `projector add` and `projector get` work through a chosen connected profile instead of a raw address.

### `projector deploy`

Guides the user through provisioning a remote server over SSH, launches the default BYO target inside a sysbox-backed container, and registers the resulting server as a named profile ready for `connect` and `sync`. The normal single-user BYO target should still be one projector-server binary plus one SQLite database file, but isolated inside the sysbox container. Postgres-oriented deployment should exist, but as the advanced lane rather than the default one-person path.

## Success criteria

v0 is successful if the owner can use it as the default private context layer in other ongoing projects for at least a week without falling back to ad hoc local-only gitignored folders.

Minimum bar:

- setup in a new repo is under five minutes
- local edits appear on another machine quickly enough to trust the system
- restart and reconnect are routine, not scary
- accidental concurrent edits do not lose content
- when something surprising happens, `status` and `log` make the state legible

## Recommended technical cut

### Bodies

- deterministic server-side three-way merge per document in v0
- conflict markers when concurrent edits overlap
- text file on disk is a materialized view of the canonical text state

This keeps the hard part of overlapping edits in the right layer now, while leaving room to replace the v0 merge strategy with Yjs later.

### Manifest

- server-backed workspace manifest with stable document ids
- path, kind, live or deleted state, and timestamps

The manifest should be authoritative for which files exist locally.

### Provenance

- append-only event log in server storage
- record actor, workspace, document id, path snapshot, event type, timestamp

This does not need a polished UI in v0. CLI inspection is enough.

### History hardening

After `0.1.0`, the next history hardening work should be:

- surgical redaction or purge of sensitive revisions by sequence
- per-sync-entry history retention and compression policy

Those are separate from restore. Restore re-applies prior state as a new head revision; redaction and retention policy are destructive history-management features.

### Server storage

Prefer the simplest durable server shape that supports multi-machine dogfooding:

- one authoritative server per sync entry
- one machine can know about multiple named server profiles
- bring-your-own server as the v0 base case
- SQLite as the default single-user BYO store
- Postgres as the advanced cloud, PaaS, or managed-service store
- guided remote deployment as the better-DX path
- a machine-global daemon stays on and reconciles bound repos continuously

The goal is operational simplicity without pretending local-only hosting solves the real cross-machine problem.

## Implementation sequence

### Slice 1: Single machine loop

- bind repo to workspace
- materialize configured mounts
- sync one text file through the server

### Slice 2: Multi-machine loop

- same workspace on two checkouts
- remote changes propagate both ways
- durable event log visible in CLI

### Slice 3: Real dogfood

- run it against several active repos
- use it with agents making normal file edits
- collect failure cases around reloads, deletes, reconnects, and path churn

## Redesign pressure

The current implementation proved out sync semantics with one checkout-level binding pointing at one workspace and one server address. The planned `add/remove` model is more granular.

That means the v0 product target now wants:

- repo projector config with one or more path-scoped sync entries
- one server-profile reference per sync entry
- one stable server-side sync-entry id per synced file or folder root
- machine-global daemon coordination across many entries, not one coarse checkout binding

The current code already has some helpful building blocks:

- manifest entries already carry `mount_relative_path`
- the materializer already understands multiple projection roots
- the machine-global daemon direction is compatible with coordinating many entries

But the current checkout-binding shape is too coarse and should be treated as redesign territory, not as the final model.

## Open design decisions

These need answers before implementation hardens:

- the exact CLI for connecting to, selecting, and inspecting server profiles
- the exact global storage shape for named server profiles and default-profile selection
- the exact guided deployment flow and how much SSH/container orchestration it should own
- the exact add/remove UX and repo projector configuration shape under the machine-global daemon model
- how local file writes are debounced and protected from daemon write loops
- how much explicit conflict or reload signaling the CLI should expose
- whether repo identity is derived from path, explicit init, or remote binding metadata
